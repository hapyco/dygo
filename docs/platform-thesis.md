# dygo Platform Thesis

## What dygo is

dygo is an opinionated framework for building serious business software in Go.

It exists for teams building business processes, internal operating systems, enterprise applications, and workflow-heavy products where security, consistency, reliability, observability, and analytics matter from day one.

The goal is not to make another lightweight starter kit. The goal is to give builders a safe, structured platform for creating business systems without reinventing the same foundations on every project.

## Why dygo exists

Agentic development is changing how software gets built.

Builders can now move faster than ever with coding agents, AI-assisted implementation, generated code, and rapid iteration. That speed is useful, and dygo should embrace it.

But serious business software cannot be built like disposable vibe-coded apps.

Business systems need predictable behavior. They need audit logs. They need permissions. They need stable data models. They need migrations, observability, logging, analytics, background jobs, secure credentials, site management, and consistent UI patterns.

Without strong boundaries, agentic coding can easily create messy, inconsistent, insecure, and hard-to-maintain systems.

dygo exists to keep the speed of agentic development while reducing the mess it can create.

## The core idea

Builders should focus on business logic.

The framework should handle everything else.

That means dygo should provide strong defaults and built-in capabilities for:

- CLI workflows
- app/module installation
- schema-driven documents
- permissions and access control
- observability
- logging
- auditing
- background jobs
- scheduling
- migrations
- site management
- maintenance mode
- configuration
- encrypted credentials
- analytics-ready data structures
- consistent Desk UI patterns
- secure APIs
- developer and agent documentation

The implementer should not spend their time rebuilding these foundations for every business application.

## Opinionated by design

dygo should have opinions on how enterprise-grade business apps are built.

It should provide safe boundaries around:

- how data is modeled
- how permissions are enforced
- how modules are installed
- how jobs are scheduled
- how logs and audit trails are captured
- how apps expose views and workflows
- how the UI should behave
- how environments and credentials are handled
- how sites are configured and maintained

These opinions are not limitations. They are guardrails.

The purpose is to let teams move fast without creating fragile systems.

## Agentic development, but with rails

dygo should actively support agentic coding.

That includes:

- clear documentation for coding agents
- well-documented CLI commands
- predictable project structure
- small, focused files
- explicit schemas and manifests
- strong naming conventions
- examples that agents can copy safely
- repeatable app-generation workflows
- agent skills for common implementation tasks

The goal is to make agentic coding useful for serious business software, not just prototypes.

Agents should be able to build on top of dygo by following strong conventions instead of guessing architecture from scratch.

## What dygo is not

dygo is not an AI slop generator.

dygo is not a vibe-coding playground.

dygo is not just an admin template.

dygo is not only for large enterprises.

dygo is for businesses of different sizes that need reliable systems around real operations.

Small teams should get enterprise-grade foundations without hiring a platform team.

Large teams should get consistency, observability, and maintainability without slowing every implementation down.

## What builders should experience

A builder using dygo should feel like the boring enterprise foundation is already handled.

They should mostly define:

- the business entities
- the business workflows
- the fields and validations
- the permissions that are specific to the domain
- the reports and dashboards the business needs
- the hooks and jobs that encode business behavior

Everything around that should feel like it just works.

## What dygo should make possible

Implementers should be able to build full operating systems for businesses using dygo.

That means not just isolated apps, but connected systems with:

- modules
- records
- workflows
- roles
- permissions
- reports
- dashboards
- file handling
- jobs
- audit trails
- site-level configuration
- consistent operational UI

The final product should feel custom to the business, but structurally clean because dygo provides the platform underneath.

## Design principle

Use agentic coding for speed.

Use dygo for structure.

The result should be business software that is fast to build, safe to operate, and clean enough to scale.
